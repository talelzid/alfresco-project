Guide — Plan de test (ISO 29119-3) : structure type, exemple commenté, bonnes pratiques

Gabarit conforme à la norme ISO/IEC/IEEE 29119-3:2013 pour préparer une campagne de test complète (Alfresco / projet QA).

Structure type d'un plan de test (ISO 29119-3)

  1. Identifiants et références
  2. Introduction
  3. Objets de test
  4. Éléments à tester
  5. Éléments non testés
  6. Approche de test
  7. Critères d'entrée / sortie / suspension / reprise
  8. Livrables de test
  9. Tâches de test
  10. Environnement de test
  11. Ressources et responsabilités

1. Identifiants et références

Informations d'identification du plan de test et références aux documents liés :

  • ID du plan : PdT-[Projet]-[Version] (ex: PdT-ALFRESCO-v1.0)
  • Version : Numéro de version du plan (ex: 1.2)
  • Date : Date de création / dernière mise à jour
  • Auteur(s) : Nom(s) des rédacteurs
  • Documents de référence : Spécifications, cahier des charges, user stories, architecture technique

2. Introduction

Le plan de test formalise l'approche qualité : périmètre, méthodes, ressources et critères permettant d'estimer si le produit est prêt.

Objectifs du plan de test :

  • Valider que les exigences fonctionnelles et non-fonctionnelles sont satisfaites.
  • Détecter et documenter les anomalies.
  • Assurer la traçabilité exigences ⇄ tests ⇄ anomalies.
  • Fournir une base objective pour la décision de mise en production.

3. Objets de test

Identifier les composants, modules ou systèmes qui seront testés (vue d'ensemble) :

  • Application : Alfresco Community 7.0
  • Modules : Gestion documentaire, recherche, workflow, permissions
  • APIs : API REST de gestion de contenu, API de recherche
  • Interfaces : Interface Web, interface mobile (si applicable)

4. Éléments à tester

Liste détaillée des fonctionnalités, modules et composants à tester priorisés :

  • Module d'upload de documents (tous formats supportés)
  • Moteur de recherche avancée (full-text, métadonnées, filtres)
  • Gestion des permissions (lecture, écriture, admin)
  • Workflows d'approbation de documents
  • APIs REST : CRUD documents, recherche, gestion utilisateurs
  • Performance : temps de réponse upload, recherche, navigation
  • Sécurité : authentification, autorisation, gestion des sessions

5. Éléments non testés

Éléments explicitement exclus du périmètre de test :

  • Composants tiers non fournis (plugins externes non standard)
  • Intégrations planifiées pour phases ultérieures (ERP, CRM)
  • Fonctionnalités dépréciées ou en fin de vie
  • Modules en développement mais non livrés dans cette release

6. Approche de test

Stratégie globale et méthodes de test utilisées :

  • Priorisation : Risk-Based Testing — focus sur les fonctionnalités critiques identifiées avec les priorités métier.
  • Types de tests :
    • Fonctionnels : validation des exigences utilisateur
    • Non-fonctionnels : performance, sécurité, utilisabilité
    • Régression : vérification de non-régression après corrections
    • Exploratoires : découverte de défauts non anticipés
  • Niveaux de test : Unitaire (dev), intégration, système, acceptation
  • Automatisation :
    • Robot Framework + Selenium webdriver pour les tests UI
    • Cucumber + Gherkin pour les tests d'acceptation
    • Postman pour les tests API
    • JMeter pour les tests de performance
    • Exécution parallèle avec Pabot

7. Critères d'entrée / sortie / suspension / reprise

7.1 Critères d'entrée

  • Spécifications validées et baseline documentaire figée.
  • Version cible livrée et déployée en environnement de test.
  • Jeux de données de test préparés et importés.
  • Cas de test revus et approuvés.
  • Environnement de test opérationnel et accessible.

7.2 Critères de sortie

  • 100% des cas de test critiques exécutés avec succès.
  • ≥ 95% des cas de test haute priorité passés.
  • Aucune anomalie bloquante ou critique non résolue.
  • Couverture des exigences ≥ 90%.
  • Rapport de test validé par le Chef QA et approuvé par le sponsor.

7.3 Critères de suspension

Conditions déclenchant la suspension immédiate des tests :

  • Défaillance critique de l'environnement (DB inaccessible, service backend indisponible, réseau HS).
  • Blocage majeur empêchant l'exécution de > 50% des cas de test.
  • Incidents de sécurité majeurs (fuite de données, accès non autorisé).
  • Taux d'échec > 30% sur les cas critiques sans correctif prévu.

7.4 Exigences de reprise

Conditions requises avant de reprendre l'exécution après suspension :

  • Incident documenté (rapport d'incident) et cause racine identifiée.
  • Correctif appliqué et vérifié (test de non-régression minimal sur zone impactée).
  • Restauration complète des environnements et validation des jeux de données (snapshot ou réexécution des scripts d'init).
  • Autorisation formelle du Chef QA pour reprise.

8. Livrables de test

Documents et artefacts produits pendant le processus de test :

  • Plan de test : Ce document (PdT-[Projet]-[Version].md)
  • Spécifications de test : Cas de test détaillés (Xray / fichiers Robot)
  • Scripts de test automatisés : Repository Git avec suites Robot Framework
  • Rapports d'exécution : Résultats quotidiens / hebdomadaires (log.html, report.html)
  • Rapports d'anomalies : Tickets Jira avec captures, logs, étapes de reproduction
  • Matrice de traçabilité : Mapping Exigences ↔ Tests ↔ Anomalies
  • Rapport final de test : Synthèse, métriques, recommandations Go/NoGo

9. Tâches de test

Liste des activités à réaliser pendant le cycle de test :

  • Analyser les exigences et identifier les scénarios de test
  • Concevoir les cas de test (manuels et automatisés)
  • Préparer les données de test et configurer les environnements
  • Développer les scripts d'automatisation
  • Exécuter les tests (manuels et automatisés)
  • Enregistrer et suivre les anomalies (cycle de vie dans Jira)
  • Effectuer les tests de régression après corrections
  • Générer et diffuser les rapports d'avancement
  • Conduire les revues de défauts et réunions de statut
  • Préparer et présenter le rapport final de test

10. Environnement de test

Configuration technique des environnements de test :

  • Environnements :
    • DEV : https://alfresco-dev.example.com (tests unitaires / intégration)
    • QA : https://alfresco-qa.example.com (tests système / acceptation)
    • STAGING : https://alfresco-staging.example.com (tests pré-production)
  • Base de données : PostgreSQL 13.x (snapshots quotidiens)
  • Navigateurs : Chrome (dernière version), Firefox, Edge
  • Système d'exploitation : Tests sur Windows 10/11, Ubuntu 20.04 LTS
  • Outils d'automatisation :
    • Robot Framework 6.x + SeleniumLibrary
    • Selenium webdriver
    • Postman / Newman pour tests API
    • JMeter 5.x pour tests de charge
  • Comptes de test : user_admin, user_contributor, user_consumer

13. Risques et contingences

Identification des risques et plans de mitigation :

Risque Probabilité Impact Mesure préventive / Contingence
Retard dans livraison d'un composant Moyenne Élevé Plan B : réduire périmètre ou mock service / Décalage planning
Jeux de données invalides Élevée Moyen Scripts de génération automatisés + snapshots réguliers DB
Montée en charge non évaluée Moyenne Élevé Tests performance précoces (JMeter) dès sprint 2
Instabilité environnement de test Élevée Élevé Infrastructure as Code (Docker), monitoring proactif
Départ membre clé de l'équipe Faible Élevé Documentation complète, binômage, formation croisée

14. Besoins en formation

Formations nécessaires pour assurer la bonne exécution et maintenance des tests :

  • Formation Xray / Jira : Gestion des cas de test (création, exécution, lien vers US, rapports) — 1 jour pour testeurs fonctionnels.
  • Atelier Robot Framework : Bonnes pratiques POM (Page Object Model), keywords réutilisables, gestion des données — 2 jours pour testeur automation.
  • Postman & API testing : Assertions automatisées, collections, environnements, intégration CI/CD — 0.5 jour pour toute l'équipe QA.
  • JMeter avancé : Tests de charge, analyse des métriques, optimisation — 1 jour pour testeur performance.

Bonnes pratiques

Recommandations pour rédiger un plan de tests efficace :

  • Être précis et concis : éviter le jargon inutile, aller à l'essentiel.
  • Traçabilité : lier chaque test à une exigence ou user story.
  • Prioriser par risque : concentrer l'effort sur les fonctionnalités critiques.
  • Définir clairement les critères : entrée, sortie, suspension et reprise.
  • Automatiser intelligemment : tests de régression, smoke tests, scénarios répétitifs.
  • Documenter les environnements : URLs, versions, comptes de test, données.
  • Prévoir les risques : identifier les blocages potentiels et les plans B.
  • Réviser régulièrement : mettre à jour le plan en fonction des évolutions du projet.
  • Impliquer l'équipe : faire valider le plan par les développeurs, PO et parties prenantes.
  • Suivre des métriques : couverture de tests, taux de détection d'anomalies, vélocité.
