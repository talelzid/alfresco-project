Guide Git & GitHub

Maîtrisez le contrôle de version et la collaboration avec Git et GitHub pour vos projets de tests automatisés.

Introduction à Git et GitHub

Git est un système de contrôle de version distribué qui permet de suivre les modifications du code source au fil du temps. GitHub est une plateforme d'hébergement de code basée sur Git qui facilite la collaboration et le partage de projets.

Pourquoi utiliser Git et GitHub dans ce projet ?

  • ✅ Historique complet : Suivre toutes les modifications du projet Alfresco
  • ✅ Collaboration : Travailler en équipe sur les tests automatisés
  • ✅ Sauvegarde : Code hébergé en ligne, accessible partout
  • ✅ GitHub Pages : Déploiement automatique du portail web
  • ✅ Gestion de versions : Revenir à une version antérieure si besoin

1. Configuration initiale de Git

A. Installation de Git

Windows :

  • Télécharger depuis git-scm.com
  • Installer avec les options par défaut
  • Vérifier l'installation : git --version

macOS :

# Via Homebrew
brew install git

# Vérifier
git --version

Linux (Ubuntu/Debian) :

sudo apt update
sudo apt install git
git --version

B. Configuration de votre identité

# Nom d'utilisateur (obligatoire)
git config --global user.name "Votre Nom"

# Email (obligatoire)
git config --global user.email "votre.email@example.com"

# Éditeur par défaut (optionnel)
git config --global core.editor "code --wait"

# Vérifier la configuration
git config --list

C. Authentification GitHub

Pour vous connecter à GitHub, vous avez deux options :

Option 1 : Personal Access Token (PAT) — recommandé

# 1. Créer un token sur GitHub
https://github.com/settings/tokens/new?scopes=repo

# 2. Copier le token (commence par ghp_)

# 3. Utiliser lors du push
Username: votre-username
Password: ghp_votre_token_ici

Option 2 : SSH Keys

# Générer une clé SSH
ssh-keygen -t ed25519 -C "votre.email@example.com"

# Copier la clé publique
cat ~/.ssh/id_ed25519.pub

# Ajouter sur GitHub : Settings > SSH and GPG keys

2. Commandes Git essentielles

Les 5 commandes de base

Commande Description Exemple
git clone Télécharger un dépôt depuis GitHub git clone https://github.com/user/repo.git
git add Préparer des fichiers pour le commit git add fichier.txt ou git add .
git commit Enregistrer les modifications git commit -m "Description"
git push Envoyer les commits vers GitHub git push origin master
git pull Récupérer les dernières modifications git pull origin master

Commandes de consultation

# Voir l'état du dépôt (fichiers modifiés, en attente)
git status

# Voir l'historique des commits
git log
git log --oneline  # Version compacte
git log --graph --all  # Voir les branches

# Voir les différences avant commit
git diff
git diff fichier.txt  # Pour un fichier spécifique

# Voir les branches
git branch  # Branches locales
git branch -a  # Toutes les branches

Commandes d'annulation

# Annuler les modifications d'un fichier (avant add)
git checkout -- fichier.txt

# Retirer un fichier du staging (après add, avant commit)
git reset fichier.txt

# Annuler le dernier commit (garder les modifications)
git reset --soft HEAD~1

# Annuler le dernier commit (perdre les modifications) ⚠️
git reset --hard HEAD~1

# Revenir à un commit précis
git checkout <hash-du-commit>

3. Workflow Git complet

Scénario A : Premier clonage du projet

# 1. Cloner le projet
git clone https://github.com/zidtalel/alfresco-project.git

# 2. Entrer dans le dossier
cd alfresco-project

# 3. Vérifier la branche actuelle
git branch

# 4. Commencer à travailler !

Scénario B : Travailler sur le projet

# 1. S'assurer d'avoir la dernière version
git pull origin master

# 2. Modifier des fichiers (ex: ajouter un test)
# ... édition de fichiers ...

# 3. Voir ce qui a changé
git status
git diff

# 4. Ajouter les fichiers modifiés
git add pages/guides/guide_github.html
# ou tout ajouter :
git add .

# 5. Créer un commit avec un message descriptif
git commit -m "Ajout du guide GitHub"

# 6. Envoyer vers GitHub
git push origin master

Scénario C : Résoudre un conflit

Un conflit se produit quand deux personnes modifient la même ligne d'un fichier.

# 1. Tentative de pull
git pull origin master
# → CONFLIT détecté !

# 2. Ouvrir le fichier en conflit
# Git marque les conflits ainsi :
<<<<<<< HEAD
Votre version locale
=======
Version de GitHub
>>>>>>> branch-name

# 3. Choisir quelle version garder (ou fusionner manuellement)
# 4. Supprimer les marqueurs Git (<<<<, ====, >>>>)
# 5. Finaliser

git add fichier-corrige.txt
git commit -m "Résolution du conflit"
git push origin master

4. Gestion des branches

Les branches permettent de travailler sur des fonctionnalités indépendantes sans impacter le code principal.

Créer et utiliser une branche

# Créer une nouvelle branche
git branch feature-nouveau-test

# Changer de branche
git checkout feature-nouveau-test

# Raccourci : créer + changer
git checkout -b feature-nouveau-test

# Voir toutes les branches
git branch

# Travailler normalement
git add .
git commit -m "Développement de la nouvelle fonctionnalité"

# Envoyer la branche sur GitHub
git push origin feature-nouveau-test

Fusionner une branche (merge)

# 1. Retourner sur la branche principale
git checkout master

# 2. Fusionner la branche
git merge feature-nouveau-test

# 3. Envoyer les modifications
git push origin master

# 4. Supprimer la branche locale (optionnel)
git branch -d feature-nouveau-test

# 5. Supprimer la branche sur GitHub (optionnel)
git push origin --delete feature-nouveau-test

5. Bonnes pratiques

Messages de commit clairs et descriptifs

✅ BON

git commit -m "Ajout du guide GitHub avec exemples de commandes"
git commit -m "Correction du bug d'affichage des annonces"
git commit -m "Refonte de la page FAQ avec recherche"

❌ MAUVAIS

git commit -m "update"
git commit -m "fix"
git commit -m "test"
git commit -m "corrections diverses"

Structure d'un bon message de commit

# Format recommandé
<type>: <sujet court (50 chars max)>

<description détaillée optionnelle>

# Types courants :
feat:     Nouvelle fonctionnalité
fix:      Correction de bug
docs:     Documentation seulement
style:    Formatage, style (sans changement de code)
refactor: Refactorisation de code
test:     Ajout de tests
chore:    Tâches de maintenance

Exemples :

git commit -m "feat: ajout d'un guide GitHub interactif"
git commit -m "fix: correction de l'erreur 404 sur menu.html"
git commit -m "docs: mise à jour du README avec instructions d'installation"

Fréquence des commits

  • ✅ Commiter régulièrement (toutes les 30 min - 1h de travail)
  • ✅ Un commit = une modification logique
  • ✅ Tester avant de commiter
  • ❌ Éviter les commits trop gros (100+ fichiers)
  • ❌ Ne jamais commiter de mots de passe ou tokens

Fichiers à ignorer (.gitignore)

Créer un fichier .gitignore à la racine du projet :

# Fichiers système

# Secrets (IMPORTANT !)
