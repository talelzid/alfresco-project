Guide — Selenium WebDriver : Concepts Fondamentaux

Création de projet, structure recommandée, configuration avec config.properties et Locators.properties, stratégies de localisation d'éléments.

Introduction à Selenium WebDriver

Selenium WebDriver est une collection open source d'API utilisée pour tester les applications web. L'outil Selenium WebDriver est utilisé pour automatiser les tests d'applications web afin de vérifier qu'elles fonctionnent comme prévu.

Il prend principalement en charge les navigateurs suivants : Firefox, Chrome, Safari et Edge, permettant d'exécuter des tests multi-navigateurs.

Langages Supportés par WebDriver

WebDriver supporte plusieurs langages de programmation :

  • Java (plus utilisé sur le marché)
  • Python (très populaire)
  • .NET
  • PHP
  • Perl
  • Ruby

Dans ce guide, nous utilisons WebDriver avec Java et IntelliJ IDEA comme IDE.

1. Structure Recommandée d'un Projet WebDriver

Une structure claire et modulaire est essentielle pour faciliter la maintenance et la collaboration. Voici l'architecture recommandée pour un projet Maven avec WebDriver et TestNG.

Dossier / Fichier Rôle
📁 selenium-project/ Racine du projet Maven
📁 src/test/java/ Code source des tests (Java)
📁 config/ Classes de configuration
📄 ConfigReader.java Charge et expose les paramètres de config.properties
📁 tests/ Classes de tests
📄 LoginTest.java Cas de test pour la fonctionnalité de login
📁 src/test/resources/ Fichiers de configuration et données
📄 config.properties Configuration (navigateur, URL, timeouts, credentials)
📄 Locators.properties Tous les sélecteurs HTML (localisateurs d'éléments)
📄 pom.xml Dépendances du projet (Selenium, TestNG)
📄 testng.xml Configuration d'exécution des tests

Organisation Recommandée

Séparez clairement les classes de configuration (config/) des classes de tests (tests/). Cela rend le projet plus maintenable et facile à naviguer à mesure qu'il grandit.

2. Configuration du Projet Maven (pom.xml)

Pour utiliser Selenium WebDriver et TestNG, vous devez ajouter les dépendances nécessaires dans le fichier pom.xml.

Dépendances Essentielles

<dependencies>
  <!-- Selenium WebDriver -->
  <dependency>
    <groupId>org.seleniumhq.selenium</groupId>
    <artifactId>selenium-java</artifactId>
    <version>4.x.x</version>
  </dependency>

  <!-- TestNG -->
  <dependency>
    <groupId>org.testng</groupId>
    <artifactId>testng</artifactId>
    <version>7.x.x</version>
    <scope>test</scope>
  </dependency>
</dependencies>

3. Le Fichier config.properties

Le fichier config.properties, placé dans src/test/resources/, permet de stocker les variables accessibles à toutes les classes. Il centralise la configuration de l'environnement de test (navigateur, URL, timeouts, credentials, etc.).

Exemple de config.properties

# Configuration du navigateur et de l'application
browser=chrome
base.url=https://demo.redmine.org

# Credentials pour les tests
test.username=admin
test.password=admin123

# Stratégies d'attente (en secondes)
implicit.wait=20
explicit.wait=30

# Configuration des rapports
screenshot.on.failure=true
report.path=test-output/reports/

Cette centralisation permet de modifier facilement les paramètres sans toucher au code des tests (par exemple, changer l'URL ou les credentials sans recompiler).

Classe ConfigReader

La classe ConfigReader charge et expose les valeurs du fichier config.properties pour une utilisation dans tout le projet. 

package config;

import java.io.FileInputStream;
import java.io.IOException;
import java.util.Properties;

public class ConfigReader {
    private static Properties properties;
    private static final String CONFIG_PATH = "src/test/resources/config.properties";

    static {
        try {
            FileInputStream fis = new FileInputStream(CONFIG_PATH);
            properties = new Properties();
            properties.load(fis);
            fis.close();
        } catch (IOException e) {
            e.printStackTrace();
            throw new RuntimeException("Impossible de charger config.properties");
        }
    }

    public static String getProperty(String key) {
        return properties.getProperty(key);
    }

    public static String getBaseUrl() {
        return properties.getProperty("base.url");
    }

    public static String getBrowser() {
        return properties.getProperty("browser");
    }
}

4. Le Fichier Locators.properties

Le fichier Locators.properties, placé dans src/test/resources/, centralise tous les localisateurs (sélecteurs) des éléments de l'interface utilisateur. Cette approche facilite grandement la maintenance : si un sélecteur change, il suffit de modifier ce fichier sans toucher aux classes de test.

Exemple de Locators.properties

Pour une application type (ex: Redmine), les localisateurs pourraient être :

# Page de Login
login.username=username
login.password=password
login.button=login
login.error=//div[@id='flash_error']

# Page d'accueil
home.admin_link=Administration
home.projects_link=Projects
home.logout_link=Déconnexion

Comment Structurer Locators.properties

Les clés doivent suivre une convention de nommage claire : [page].[element]

  • login.username : élément username sur la page de login
  • login.password : élément password sur la page de login
  • home.admin_link : lien Administration sur la page d'accueil

Avantage Principal

La séparation des localisateurs du code permet aux testeurs de maintenir les sélecteurs sans connaître la programmation Java.

5. Stratégies de Localisation d'Éléments

Avant de coder la méthode de test, il est nécessaire de préparer les localisateurs des éléments. L'instruction findElement de WebDriver permet de localiser un élément avec différentes stratégies.

Syntaxe Générale

driver.findElement(By.LocatorStrategy("LocatorValue"));

Les Stratégies de Localisation Disponibles

Stratégie Syntaxe Exemple Utilisation
id By.id() By.id("username") Localise par l'attribut id (recommandé si disponible)
name By.name() By.name("login") Localise par l'attribut name
linkText By.linkText() By.linkText("Administration") Localise un lien par son texte exact
cssSelector By.cssSelector() By.cssSelector("input[type='submit']") Utilise les sélecteurs CSS
xpath By.xpath() By.xpath("//button[@id='submit']") Utilise XPath (très flexible mais plus lent)

5.1. Localisation par ID

driver.findElement(By.id("username")).sendKeys("admin");

C'est la stratégie la plus recommandée car l'ID doit être unique sur une page et est généralement stable.

5.2. Localisation par Name

driver.findElement(By.name("password")).sendKeys("password123");

5.3. Localisation par linkText

driver.findElement(By.linkText("Administration")).click();

Utilisé spécifiquement pour localiser un lien hypertexte avec son texte exact.

5.4. Localisation par CSS Selector

driver.findElement(By.cssSelector("input[type='submit']")).click();
driver.findElement(By.cssSelector(".btn-primary")).click();
driver.findElement(By.cssSelector("#submit-button")).click();

5.5. Localisation par XPath

driver.findElement(By.xpath("//input[@id='username']")).sendKeys("admin");
driver.findElement(By.xpath("//button[text()='Login']")).click();
driver.findElement(By.xpath("//a[contains(@href, 'administration')]")).click();

XPath est très flexible mais plus lent que les autres stratégies. À utiliser quand les autres méthodes ne suffisent pas.

Ordre de Préférence

1. ID (le plus recommandé) → 2. Name → 3. CSS Selector → 4. XPath (dernier recours)

6. Exemple Complet : Test de Connexion

Voici un exemple complet d'une classe de test qui démontre l'utilisation de config.properties, Locators.properties et les stratégies de localisation. 

package tests;

import org.openqa.selenium.WebDriver;
import org.openqa.selenium.By;
import org.openqa.selenium.chrome.ChromeDriver;
import org.testng.annotations.*;
import config.ConfigReader;
import java.io.FileInputStream;
import java.util.Properties;

public class LoginTest {
    private static WebDriver driver;
    private static Properties locators;

    @BeforeClass
    public static void setUpBeforeClass() {
        // Instancier le driver
        driver = new ChromeDriver();
        
        // Charger le fichier Locators.properties
        try {
            FileInputStream fis = new FileInputStream(
                "src/test/resources/Locators.properties"
            );
            locators = new Properties();
            locators.load(fis);
            fis.close();
        } catch (Exception e) {
            e.printStackTrace();
        }
        
        // Maximiser la fenêtre
        driver.manage().window().maximize();
        
        // Fixer un timeout implicite de 20 secondes
        driver.manage().timeouts().implicitlyWait(
            java.time.Duration.ofSeconds(20)
        );
    }

    @Test
    public void testLogin() {
        // Récupérer l'URL de base et les localisateurs
        String baseUrl = ConfigReader.getBaseUrl();
        String username = ConfigReader.getProperty("test.username");
        String password = ConfigReader.getProperty("test.password");
        
        String usernameLoc = locators.getProperty("login.username");
        String passwordLoc = locators.getProperty("login.password");
        String buttonLoc = locators.getProperty("login.button");
        
        // Ouvrir l'application
        driver.get(baseUrl);
        
        // Saisir le username
        driver.findElement(By.id(usernameLoc)).sendKeys(username);
        
        // Saisir le password
        driver.findElement(By.id(passwordLoc)).sendKeys(password);
        
        // Cliquer sur le bouton login
        driver.findElement(By.name(buttonLoc)).click();
        
        // Naviguer vers Administration
        driver.findElement(
            By.linkText(locators.getProperty("home.admin_link"))
        ).click();
        
        // Déconnexion
        driver.findElement(
            By.linkText(locators.getProperty("home.logout_link"))
        ).click();
    }

    @AfterClass
    public static void tearDownAfterClass() {
        // Quitter le driver
        driver.quit();
    }
}

7. Isolation des Tests

Il est important de décider si vos tests s'exécutent avec ou sans isolation :

Sans Isolation (SharedWebDriver)

  • ✅ Avantage : Exécution plus rapide (un navigateur pour tous les tests)
  • ❌ Inconvénient : Risque d'effets de bord entre les tests

Avec Isolation (IndividualDriver)

  • ✅ Avantage : Tests totalement indépendants, plus robustes
  • ❌ Inconvénient : Plus lent (relance du navigateur à chaque test)

Pour l'isolation, utilisez les annotations @BeforeMethod et @AfterMethod à la place de @BeforeClass et @AfterClass. Cela garantit qu'un nouveau driver est créé avant chaque test et fermé après.

Bonnes Pratiques Essentielles

  • Centraliser la configuration : Utilisez config.properties pour tous les paramètres.
  • Centraliser les localisateurs : Utilisez Locators.properties pour tous les sélecteurs.
  • Privilégier ID : Utilisez l'attribut ID en priorité s'il est disponible.
  • Éviter les XPath absolus : Préférez les XPath relatifs et plus spécifiques.
  • Nommage clair : Utilisez des conventions cohérentes pour les clés de config et locators.
  • Timeouts appropriés : Définissez un timeout implicite pour éviter les délais d'attente trop longs.
  • Isolation des tests : Décidez si vos tests doivent être isolés les uns des autres.

Pré-requis Techniques

  • Java JDK 11 ou supérieur
  • Maven 3.6+
  • IntelliJ IDEA Community Edition
  • Navigateur Chrome avec ChromeDriver
  • Selenium WebDriver 4.x
  • TestNG 7.x

Convention de Nommage

Pour Locators.properties, utilisez toujours la convention [pageName].[elementName] pour une lisibilité optimale et une maintenance facile.
