Guide — Framework Cucumber pour le BDD

Automatisation des tests comportementaux avec Cucumber, Gherkin et les Step Definitions.

Introduction au Behavior-Driven Development (BDD)

Le BDD (Behavior-Driven Development) est une méthodologie qui encourage la collaboration entre développeurs, testeurs et parties prenantes non techniques. L'objectif est de créer une compréhension commune du comportement attendu d'un système avant son implémentation.

Cucumber est un framework open-source qui facilite le BDD en permettant d'écrire des tests dans un langage naturel (Gherkin) qui peut être lu et compris par tous les membres de l'équipe.

Pourquoi utiliser Cucumber ?

  • Documentation vivante : Les scénarios Gherkin servent de documentation toujours à jour
  • Collaboration améliorée : Langage commun entre l'équipe technique et métier
  • Tests automatisés : Exécution automatique des scénarios via le code
  • Traçabilité : Lien direct entre exigences métier et tests

Architecture de Cucumber

Cucumber suit une architecture en trois couches qui sépare les responsabilités :

Composant Rôle Format
Feature Files Fichiers contenant les scénarios de test écrits en Gherkin .feature
Step Definitions Code qui implémente les étapes Gherkin (Given, When, Then) .java, .py, .rb, etc.
Test Runner Orchestre l'exécution des tests et génère les rapports Classe Java annotée ou configuration

Flux d'exécution de Cucumber

1. Le Test Runner lit les fichiers .feature
2. Cucumber analyse les scénarios Gherkin
3. Pour chaque étape (Given/When/Then), Cucumber cherche la Step Definition correspondante
4. Le code de la Step Definition est exécuté
5. Les résultats sont collectés et un rapport est généré

Structure d'un projet Cucumber

Organisation typique des fichiers dans un projet Cucumber :

src/
├── test/
│   ├── java/
│   │   ├── runners/

Les Step Definitions

Les Step Definitions sont le pont entre le langage Gherkin et le code d'automatisation. Chaque étape Gherkin est mappée à une méthode Java (ou autre langage) via des annotations.

Annotations principales

Annotation Usage Exemple
@Given Préconditions et contexte initial @Given("l'utilisateur est sur la page de connexion")
@When Actions effectuées par l'utilisateur @When("il entre ses identifiants valides")
@Then Vérifications et assertions @Then("il doit être redirigé vers le tableau de bord")
@And / @But Continuation d'étapes (même annotation que la précédente) @And("un message de bienvenue s'affiche")

Exemple de Step Definition

public class LoginSteps {
    WebDriver driver;
    
    @Given("l'utilisateur est sur la page de connexion")
    public void utilisateur_sur_page_connexion() {
        driver = new ChromeDriver();
        driver.get("https://redmine.example.com/login");
    }
    
    @When("il entre le nom d'utilisateur {string} et le mot de passe {string}")
    public void entre_identifiants(String username, String password) {
        driver.findElement(By.id("username")).sendKeys(username);
        driver.findElement(By.id("password")).sendKeys(password);
        driver.findElement(By.name("login")).click();
    }
    
    @Then("il doit être connecté avec succès")
    public void verifie_connexion_reussie() {
        Assert.assertTrue(driver.findElement(By.className("loggedinas")).isDisplayed());
        driver.quit();
    }
}

Paramètres dans les Step Definitions

Cucumber supporte plusieurs types de paramètres pour rendre les steps réutilisables :

  • Strings : {string} pour capturer du texte entre guillemets
  • Integers : {int} pour les nombres entiers
  • Floats : {float} ou {double} pour les décimaux
  • Expressions régulières : Pour des patterns complexes
@When("il crée {int} nouveaux tickets")
public void cree_plusieurs_tickets(int nombre) {
    for (int i = 0; i < nombre; i++) {
        // Logique de création
    }
}

Les Hooks

Les Hooks permettent d'exécuter du code avant et après les scénarios ou les steps. Ils sont utilisés pour la configuration et le nettoyage.

Hook Moment d'exécution Usage typique
@Before Avant chaque scénario Initialisation du WebDriver, connexion à la base de données
@After Après chaque scénario Fermeture du navigateur, capture d'écran en cas d'échec
@BeforeStep Avant chaque étape Logging détaillé
@AfterStep Après chaque étape Vérifications intermédiaires

Exemple de Hooks

public class Hooks {
    WebDriver driver;
    
    @Before
    public void setUp() {
        System.out.println("Démarrage du scénario...");
        driver = new ChromeDriver();
        driver.manage().timeouts().implicitlyWait(10, TimeUnit.SECONDS);
    }
    
    @After
    public void tearDown(Scenario scenario) {
        if (scenario.isFailed()) {
            // Capture d'écran en cas d'échec
            byte[] screenshot = ((TakesScreenshot) driver).getScreenshotAs(OutputType.BYTES);
            scenario.attach(screenshot, "image/png", "Échec du scénario");
        }
        driver.quit();
        System.out.println("Scénario terminé.");
    }
}

Configuration du Test Runner

Le Test Runner est une classe qui configure l'exécution des tests Cucumber. Il utilise des annotations pour spécifier les options. 

import org.junit.runner.RunWith;
import io.cucumber.junit.Cucumber;
import io.cucumber.junit.CucumberOptions;

@RunWith(Cucumber.class)
@CucumberOptions(
    features = "src/test/resources/features",           // Chemin des fichiers .feature
    glue = {"steps", "hooks"},                          // Packages des step definitions
    plugin = {"pretty", "html:target/cucumber-reports"}, // Formats de rapport
    tags = "@smoke",                                     // Tags à exécuter
    monochrome = true,                                   // Sortie console lisible
    dryRun = false                                       // true = vérification sans exécution
)
public class TestRunner {
}

Options importantes du Test Runner

Option Description
features Chemin vers les fichiers .feature
glue Packages contenant les step definitions et hooks
plugin Formats de rapport (HTML, JSON, XML, etc.)
tags Filtrage des scénarios par tags (@smoke, @regression, etc.)
dryRun Vérifier la correspondance entre steps et définitions sans exécuter

Les Tags dans Cucumber

Les tags permettent d'organiser et de filtrer l'exécution des scénarios. Ils sont préfixés par @ et placés avant les Features ou Scenarios. 

@regression @login
Feature: Authentification utilisateur

  @smoke @critical
  Scenario: Connexion avec des identifiants valides
    Given l'utilisateur est sur la page de connexion
    When il entre des identifiants valides
    Then il doit être connecté avec succès
    
  @negative
  Scenario: Connexion avec des identifiants invalides
    Given l'utilisateur est sur la page de connexion
    When il entre des identifiants invalides
    Then un message d'erreur doit s'afficher

Combinaison de tags

  • @smoke : Exécute tous les scénarios tagués @smoke
  • @smoke and @critical : Scénarios ayant les deux tags
  • @smoke or @regression : Scénarios ayant l'un ou l'autre
  • not @wip : Exclut les scénarios en "Work In Progress"

Scenario Outline

Les Scenario Outlines permettent d'exécuter le même scénario avec différents jeux de données. 

Scenario Outline: Vérifier la recherche de tickets
  Given je suis connecté en tant que "<Utilisateur>"
  When je recherche les tickets avec le statut "<Statut>"
  Then je dois voir <Nombre> tickets

  Examples:
    | Utilisateur | Statut   | Nombre |
    | Admin       | Nouveau  | 15     |
    | Manager     | En cours | 8      |
    | Testeur     | Fermé    | 32     |

Bonnes pratiques Cucumber

Conventions de nommage

Fichiers .feature : Noms descriptifs en minuscules avec underscores (ex: user_login.feature)
Step Definitions : Noms de classe en PascalCase avec suffixe "Steps" (ex: LoginSteps.java)

  • Une action par scénario : Chaque scénario doit tester un comportement spécifique
  • Steps réutilisables : Évitez la duplication en créant des steps génériques
  • Indépendance des scénarios : Chaque scénario doit pouvoir s'exécuter seul
  • Background pour la répétition : Utilisez Background pour les préconditions communes
  • Tags stratégiques : Organisez vos tests avec des tags significatifs (@smoke, @regression, @api)
  • Pas de logique dans les features : Gardez Gherkin simple et lisible
  • Page Object Model : Combinez Cucumber avec le pattern POM pour une meilleure maintenabilité

Pièges à éviter

❌ Ne pas dupliquer le code dans les step definitions
❌ Éviter les steps trop génériques (ex: "je clique sur le bouton")
❌ Ne pas tester l'implémentation technique dans les scénarios
❌ Éviter les dépendances entre scénarios

Rapports et Intégration Continue

Cucumber génère plusieurs formats de rapports pour faciliter l'analyse des résultats :

  • HTML : Rapport visuel détaillé avec statuts des scénarios
  • JSON : Format structuré pour l'intégration avec d'autres outils
  • JUnit XML : Compatible avec les outils CI/CD (Jenkins, GitLab CI)
  • Pretty Console : Sortie formatée dans la console

Pour intégrer Cucumber dans un pipeline CI/CD, utilisez Maven ou Gradle pour automatiser l'exécution et la génération des rapports : 

mvn clean test -Dcucumber.options="--tags @smoke"
